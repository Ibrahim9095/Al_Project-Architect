# AI Project Architect

# Engineering Standard

## 1. Purpose

This document defines the engineering standards, documentation principles, repository rules, and quality requirements that govern every project created using AI Project Architect.

It establishes a unified Software Engineering methodology to ensure consistency, traceability, maintainability, and professional documentation across every generated repository.

Every document, prompt, template, schema, and generated artifact must comply with the standards defined in this document.

This document serves as the constitutional foundation of the entire platform.

---

# 2. Engineering Philosophy

AI Project Architect follows a Documentation-First Software Engineering methodology.

Software implementation is considered the final engineering activity—not the first.

Every engineering decision must originate from validated business knowledge.

The platform follows the following engineering lifecycle:

```
Understand

↓

Discover

↓

Analyze

↓

Validate

↓

Document

↓

Architect

↓

Review

↓

Generate

↓

Implement

↓

Test

↓

Deploy
```

Skipping any stage introduces unnecessary engineering risk and is therefore prohibited.

---

# 3. Core Engineering Principles

Every project generated by AI Project Architect must follow these principles.

## Principle 1 — Business Before Technology

Technology never defines the project.

Business requirements define the technology.

---

## Principle 2 — Documentation Before Implementation

No implementation may begin before documentation has been completed and validated.

---

## Principle 3 — No Assumptions

The system must never invent:

- Business requirements
- Business rules
- User permissions
- Workflow behavior
- Technical constraints

When information is missing, the AI must request clarification.

---

## Principle 4 — Single Source of Truth

Every engineering decision must exist in exactly one place.

Documentation becomes the authoritative source for:

- Requirements
- Business Rules
- Database Design
- APIs
- Architecture
- Development Tasks

Implementation must always follow documentation.

---

## Principle 5 — Traceability

Every engineering artifact must be traceable back to its origin.

Every feature should reference:

- Business Goal
- Requirement
- Business Rule
- Module
- Database Object
- API Endpoint
- Test Case

Nothing should exist without justification.

---

## Principle 6 — Separation of Responsibilities

Every document has one responsibility.

Documents must never duplicate information already owned by another document.

For example:

Vision explains why.

Requirements explain what.

Business Rules explain how.

Database explains where data lives.

API explains communication.

Tasks explain implementation work.

---

## Principle 7 — Engineering Before Coding

AI Project Architect is an engineering platform.

Its responsibility is to prepare software for implementation.

Generating code without engineering documentation is considered an incomplete engineering process.

# 6. Documentation Standards

Documentation is the foundation of every engineering decision.

Each document must be written as an independent engineering artifact with a clearly defined responsibility.

Documentation should be:

- Accurate
- Complete
- Consistent
- Traceable
- Versionable
- Maintainable
- Human-readable
- AI-readable

Documentation must never contain assumptions.

If information is unavailable, the documentation should explicitly identify the missing information rather than attempting to infer it.

Every document should answer one primary engineering question.

For example:

- Vision explains why the project exists.
- Project Scope defines what will and will not be built.
- Requirements describe what the system must do.
- Business Rules describe how the business operates.
- Modules describe how functionality is organized.
- Database describes how information is stored.
- API describes how systems communicate.

Each document owns its own responsibility.

# 8. Repository Standards

Every repository generated by AI Project Architect must follow the same engineering conventions.

Repositories must be:

- Modular
- Predictable
- Consistent
- Scalable
- Maintainable

The repository structure should communicate the engineering architecture without requiring additional explanation.

Every directory must have a clearly defined purpose.

Generated repositories should include only the engineering assets required by the analyzed project.

Unused documentation, templates, prompts, or engineering resources should never be included.

The repository should remain clean, organized, and easy to navigate regardless of project size.

---

# 9. Naming Conventions

Consistency in naming is essential for long-term maintainability.

The following conventions apply throughout the repository.

## Documents

Documentation files should use uppercase names with numeric prefixes.

Examples:

- 00_ENGINEERING_STANDARD.md
- 01_VISION.md
- 05_REQUIREMENTS.md
- 11_API.md

Numeric prefixes define reading order and engineering dependencies.

---

## Directories

Directory names should use lowercase letters.

Examples:

- docs
- prompts
- templates
- schemas
- knowledge
- generators

Directory names should describe responsibilities rather than implementation details.

---

## Markdown Headings

Headings should form a logical hierarchy.

Use:

- H1 for document title.
- H2 for major engineering sections.
- H3 for subsections.
- H4 only when additional structure is required.

Avoid unnecessary heading depth.

# 12. Requirement Identification Standard

Every requirement within a project must have a unique identifier.

Requirement IDs provide traceability across documentation, implementation, testing, and maintenance.

Recommended format:

REQ-001
REQ-002
REQ-003

Requirements must never be duplicated.

Each requirement should have only one authoritative definition.

---

# 13. Business Rule Identification Standard

Business Rules must also have unique identifiers.

Recommended format:

BR-001
BR-002
BR-003

Every Business Rule should reference one or more related requirements whenever applicable.

Business Rules must describe business behavior rather than technical implementation.

---

# 14. Module Identification Standard

Every software module must have a unique identifier.

Recommended format:

MOD-001
MOD-002
MOD-003

Each module should clearly specify:

- Purpose
- Responsibilities
- Dependencies
- Related Requirements
- Related Business Rules

Modules should remain cohesive and independently maintainable.

---

# 15. Database Object Standards

Database documentation should describe logical data structures rather than implementation-specific details.

Each entity should define:

- Entity Name
- Description
- Primary Key
- Attributes
- Relationships
- Constraints
- Related Requirements
- Related Business Rules

Database documentation must remain independent from any specific database engine whenever possible.

---

# 16. API Documentation Standards

API documentation must define system communication clearly.

Each endpoint should include:

- Endpoint
- Method
- Description
- Request Parameters
- Request Body
- Response Body
- Status Codes
- Authentication
- Authorization
- Validation Rules
- Related Requirements
- Related Business Rules

API documentation should focus on behavior rather than implementation.

# 18. Documentation Validation

Before documentation is considered complete, it should satisfy the following validation criteria.

Completeness

- No major information is missing.
- All required sections are documented.

Consistency

- No conflicting statements exist.
- Terminology remains consistent.

Accuracy

- Documentation reflects approved business requirements.

Traceability

- Every engineering artifact references its origin.

Readability

- Documents are understandable by both humans and AI agents.

Maintainability

- Documents can be updated independently without affecting unrelated artifacts.

---

# 19. Versioning Standards

Documentation evolves together with the project.

Every significant engineering change should update the corresponding documentation.

Major changes include:

- Business scope modifications
- New requirements
- New business rules
- Architecture changes
- Database redesign
- API changes
- Security updates

Documentation versions should always remain synchronized with repository versions.

---

# 20. Change Management

Changes should never be introduced without analysis.

Every proposed change should answer:

- Why is the change required?
- What problem does it solve?
- Which documents are affected?
- Which requirements are affected?
- Which business rules are affected?
- Which modules are affected?
- Does the database change?
- Does the API change?
- Are new test cases required?

Engineering changes should always be documented before implementation.
